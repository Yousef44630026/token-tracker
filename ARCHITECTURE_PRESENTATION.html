Conception Method

How the architecture was conceived before talking about modules.

Start from the accounting problem

The design begins with the fact that token payloads are not naturally comparable. Different providers expose different fields, nested details, totals, cache values, reasoning values, and sometimes missing usage. So the first job is not storage or UI. The first job is defining what a token fact means.

Design the target behavior

The deck describes how the system should work conceptually. It does not assume the current project implementation is perfect. The conception separates the intended model from code reality, then uses tests and implementation phases to close the gap.

  1. Identify what questions the tracker must answer.
  2. Define invariants that must never be broken.
  3. Build a neutral data model around those invariants.
  4. Place boundaries where provider behavior can change.
  5. Choose storage and metrics that preserve auditability.
  6. Turn the architecture into an implementation roadmap.

Design Questions

These questions shaped every major decision.

What is observed?

A provider response, stream finish, proxy response, Codex log line, embedding call, rerank call, or collector event.

What is trusted?

Only categorized facts with known provider semantics should contribute to totals. Unknown data is preserved but excluded.

What is connected?

Events are connected by trace, span, service, model, provider, workflow, and phase so totals explain where usage came from.

What is derived?

Dashboards, exports, totals, mismatch checks, exactness ratios, and workflow metrics are derived from stored source facts.

The architecture is basically the answer to one question: how can we keep usage facts provider-neutral, auditable, privacy-safe, and impossible to double count?
Invariant First Design

The design was built around rules that must stay true forever.

Store facts

Raw provider usage is converted into source facts. Stored events should remain stable even if analytics formulas improve later.

Do not guess totals

A quantity with unknown precision or unknown additivity is not counted as zero. It is preserved and flagged as excluded.

Derive safely

Totals are computed only from contribution-aware columns such as quantity_in_total and event_contributing_tokens.

Additivity Rationale

Why additivity is a first-class contract.

The danger

Token payloads often include both detailed fields and total fields. For example, input, output, cached input, reasoning output, and provider total may not all be independent. Summing everything can turn one event into two or three events mathematically.

The design choice

Every quantity carries an additivity status. Contributing quantities can be summed. Subtotals are preserved for reconciliation. Unverified quantities are preserved but excluded until a provider-specific rule is approved.

event_contributing_tokens = 0 if event is superseded or non-authoritative else sum(quantity.amount where quantity.additivity == "contributing") provider_total_tokens = reconciliation fact only, never the default aggregate

Implementation Rationale

Why the build plan follows this order.

1. Contract

Define schemas, token categories, additivity states, and quality flags first.

2. Adapters

Map provider payloads into the contract and prove semantics with fixtures.

3. Normalizer

Create events, IDs, trace context, precision, reconciliation, and exclusions.

4. Storage

Persist source facts, then derive safe totals and exports from those facts.

5. Surfaces

Add streaming, collector, proxy, dashboard, RAG, and agent integrations.

This order reduces risk because UI and dashboards are only meaningful after the accounting contract is correct.
Conception Summary

The architecture is a chain of deliberate tradeoffs.

What the design optimizes for

  • Correct accounting before attractive reporting.
  • Provider-neutral facts before provider-specific dashboards.
  • Traceability before aggregate convenience.
  • Privacy-safe observability before content-heavy logging.

What the design intentionally avoids

  • One flat token total that hides uncertainty.
  • Provider-specific logic spread across the project.
  • Counting unknown or unverified quantities silently.
  • Building dashboards before proving aggregation rules.
That is the core conception: build a small accounting language for AI usage, then let every service, metric, dashboard, and implementation phase speak that same language.

Problem

AI token accounting breaks when raw usage fields are treated as totals.

Provider payloads differ

OpenAI, Anthropic, Gemini, Bedrock, Cohere, Mistral, and rerank/embedding APIs expose different usage shapes and field names.

Fields have different semantics

Some fields are additive buckets, some are subtotals, some are estimates, and some are unverified until proven by real payload behavior.

Operational reality is messy

Streams interrupt, retries happen, provider APIs drift, usage fields disappear, and instrumentation must not crash production calls.

Architecture risk: if dashboards sum raw token fields or provider totals, they can double count cached tokens, reasoning tokens, modality breakdowns, retries, and partial stream estimates.
Promise

The tracker should be a normalization and provenance layer, not a billing engine.

It should guarantee

  • each provider observation becomes a structured event
  • each token quantity has a type, precision, source, and additivity
  • provider totals reconcile event facts but are not aggregate totals
  • retries and superseded streams contribute zero
  • unknown counts are represented as unknown, not hidden as zero

It should enable

  • auditable event-level facts
  • safe trace rollups
  • CSV/Excel exports with summable columns
  • provider drift detection
  • privacy-preserving proxy and live usage reporting

Counting Contract

The only safe total is the sum of contribution-aware quantities.

quantity_in_total = quantity when additivity == "total_contributing" and quantity is known otherwise 0 event_contributing_tokens = 0 when event is superseded 0 when observation.authoritative == false otherwise sum(quantity.quantity_in_total) trace_total = sum(event.event_contributing_tokens for event in trace.events)

Safe to sum

  • quantity_in_total at quantity grain
  • event_contributing_tokens at event grain

Never sum directly

  • raw quantity
  • provider_total_tokens across events
Additivity

Every token field must declare whether it counts, decorates, or waits for proof.

total_contributing

The field is an additive bucket. Examples: plain input, plain output, embeddings, rerank input, Gemini thinking, Anthropic cache buckets.

subtotal_of

The field is a breakdown of a parent. Examples: OpenAI cached input, OpenAI reasoning, OpenAI audio input/output, Gemini image/audio/video details.

unverified

The field is visible but excluded. It contributes zero and raises a flag until real semantics are proven. This is the fail-closed path.

Additivity should come from a provider/surface truth table. It should never be inferred from token type text alone.
Counting Rules Deep Dive

Counting is a staged decision, not a single sum.

1. Quantity rule
Decide whether each token quantity is known, exact enough, and additive.
2. Event rule
Decide whether the whole observation is authoritative and not superseded.
3. Trace rule
Roll up authoritative event contributions within a workflow trace.
4. Grouping rule
Group safe contributions by provider, model, service, span, workflow, or day.
5. Reconcile
Compare safe contribution totals to provider totals without summing provider totals.
The tracker should never ask "how many tokens are in this JSON?" It should ask "which observed quantities are allowed to contribute to which aggregate?"
Quantity Inclusion Rule

A quantity contributes only when it passes every gate.

quantity_is_countable = quantity.value is not null and quantity.precision in {"exact", "provider_reported", "accepted_estimate"} and quantity.additivity == "total_contributing" and quantity.excluded_reason is null quantity_in_total = quantity.value if quantity_is_countable else 0
Gate Passes when... Fails when...
PresenceThe provider or tracker has a numeric value.The value is missing, null, or not parseable.
PrecisionThe value is exact or explicitly accepted as an estimate for that report.The value is unknown, partial, provisional, or locally guessed without acceptance.
AdditivityThe field is a proven additive bucket.The field is a subtotal, duplicate, provider total, or unverified field.
ExclusionNo rule has excluded the quantity.Provider drift, parse warning, policy rule, or invalid semantic mapping excludes it.

Event Inclusion Rule

Even valid quantities do not count if the event itself is not authoritative.

event_is_countable = event.superseded == false and event.observation.authoritative != false and event.observation.status in {"complete", "accepted_estimate"} and event.event_type not in {"diagnostic", "dry_run", "preview"} event_contributing_tokens = sum(quantity.quantity_in_total for quantity in event.quantities) if event_is_countable else 0

Counts

A completed provider response with normalized additive quantities and no supersession.

Visible but excluded

Partial stream estimate, failed proxy request, non-authoritative probe, dry run, or superseded retry attempt.

Never deleted silently

Excluded events stay queryable with exclusion reasons so reliability and drift metrics still work.

Retry and Supersession Counting Rule

Only the winning attempt should contribute to the workflow total.

Attempt 1
Timeout after partial response. Event remains stored with superseded=true.
Attempt 2
Retry completes. Event is authoritative and contributes.
Trace rollup
Counts attempt 2 only; attempt 1 remains visible as reliability evidence.
Operations metric
Retry count, wasted estimate, and failed attempt rate can still be reported.

Trace and Service Rollup Rule

Rollups group safe event contributions; they do not recalculate provider usage.

trace_total = sum(event_contributing_tokens where event.trace_id == trace_id) service_total = sum(event_contributing_tokens where event.service_name == service_name) span_total = sum(event_contributing_tokens where event.span_id == span_id) workflow_total = sum(event_contributing_tokens where event.workflow == workflow_name)

Good group-by dimensions

trace_id, span_id, service_name, workflow, provider, model, api_surface, day.

Bad group-by source

Do not group raw provider totals or raw quantities. They are not guaranteed to be additive across rows.

Counting Implementation Checklist

The implementation should make wrong totals hard to create.

Adapter responsibilities

  • Map every known provider field to one accounting role.
  • Mark unrecognized token fields as unverified.
  • Store provider total separately from quantities.
  • Attach adapter/provider/API version metadata.

Normalizer responsibilities

  • Compute quantity_in_total only from approved rules.
  • Set event authority, status, supersession, and exclusion reasons.
  • Compute reconciliation status without using provider total as rollup.
  • Emit warnings for unknown, unverified, partial, or mismatched observations.
Tests should assert both positive totals and negative exclusions: the dangerous bugs are usually fields that look reasonable but should not be counted.

Precision Model

Token type, precision, and source are deliberately separate axes.

What is counted

  • input
  • output
  • cached_input
  • reasoning
  • thinking
  • embedding
  • rerank_input

How reliable it is

  • exact: provider-reported or authoritative
  • estimate: local tokenizer or forecast
  • unknown: count is missing and must stay visible

Where it came from

  • provider response
  • provider stream final
  • partial stream tokenizer
  • local tokenizer
  • historical forecast
  • none

This prevents bad states like estimated_output as a fake token type. The token is still output; only precision changes.

Adapter Boundary

Adapters are the provider-specific translation boundary.

Adapter should do

  1. locate the provider usage object
  2. extract documented token fields
  3. map fields to neutral token types
  4. assign precision and source
  5. assign additivity and parent subtotal
  6. preserve raw provider total
  7. emit adapter-owned flags like raw_usage_missing

Adapter should not do

  • compute trace totals
  • persist events
  • set supersession
  • hide unknown fields as zero
  • invent provider totals
  • silently ignore token-looking fields without an allowlisted reason
Adapter Creation Process

We create adapters by turning provider behavior into an explicit translation contract.

Study payloads
Collect documented fields, real examples, streaming endings, errors, and missing-usage cases.
Classify fields
Decide token type, precision, source, additivity, parent subtotal, and provider total.
Encode mapping
Implement extraction with defensive reads and explicit handling for absent fields.
Validate
Use fixtures to prove normal, missing, drifted, subtotal, and mismatch cases.
Govern
Version adapter semantics and flag unknown token-like fields until reviewed.
The adapter is not just a parser. It is the place where provider-specific meaning becomes a neutral accounting fact.

Adapter Output Contract

Adapters produce neutral usage facts, not final events.

Adapter output should contain

  • normalized token quantities
  • provider total if available
  • provider request/response identifiers when visible
  • adapter semantic version
  • extraction warnings and unknown-field flags

Adapter output should avoid

  • trace totals
  • workflow labels
  • collector decisions
  • dashboard formatting
  • pricing or billing policy
This boundary keeps adapters testable: give them a raw payload, expect a deterministic set of neutral usage facts.

Non-Streaming Lifecycle

Detailed end-to-end path for a normal provider response.

  1. User or scheduler starts a business workflow.
  2. Application opens a root trace with business labels.
  3. Application opens a span for the model-facing unit of work.
  4. Application calls the provider API.
  5. Provider returns a response with a provider-specific usage payload.
  6. The correct adapter is selected by provider and API surface.
  7. Adapter extracts usage fields and builds normalized quantities.
  8. Normalizer merges adapter facts with active context and quality flags.
  9. One event is returned to the application.
  10. Application attaches the event to the trace.
  11. Event is appended to local JSONL or trace snapshot if configured.
  12. Event is recorded by the collector client if configured.
  13. Reports derive totals from quantity_in_total and event_contributing_tokens.

Retry and Supersession

Retries and partial streams must be observable without being double counted.

Retry identity

A retry should normally keep the same trace and span, but receive a new request correlation id. This distinguishes one logical work unit from one provider attempt.

Supersession

When a later event replaces an earlier observation, the earlier event remains stored but contributes zero.

Auditability

Keeping superseded events preserves what happened operationally without polluting final totals.

if event.superseded: event_contributing_tokens = 0 else: event_contributing_tokens = sum(quantity_in_total)

Storage

Persistence should be boring, append-only, and source-fact only.

Local event store

  • append-only JSONL
  • one event per line

  • concurrent writer lock
  • crash-truncated tail recovery
  • optional durable flush

Trace snapshot store

  • atomic complete-trace snapshot
  • spans and events together
  • identity validation on load
  • no stored rollup totals

Agent Integration

Agent workflows need hierarchical spans so model calls are not flattened.

Typical agent trace

  1. root user task
  2. planner model call
  3. tool execution span
  4. retrieval or browser span
  5. sub-agent delegation span
  6. final answer model call

Important distinction

Tool spans provide structure and timing. They emit token events only when the tool itself calls a token-reporting provider. This keeps trace shape and token accounting separate.

Proxy Mode

The proxy is for observation when direct instrumentation is not possible.

Intercept

Receive local model traffic and forward it to the provider without becoming the business logic owner.

Privacy

Token observability should not become prompt observability by accident.

Store by default

  • token facts
  • provider ids when supplied
  • request/response hashes
  • prompt fingerprints
  • timing and status metadata
  • quality flags

Avoid by default

  • raw prompts
  • raw completions
  • authorization headers
  • API keys
  • full provider payloads in telemetry stores

Privacy should be verifiable with audit commands and tests, not merely promised in documentation.

Drift Defense

Provider API changes must create visible flags, not silent wrong totals.

Drift scenario

  • usage object disappears
  • field is renamed
  • new token field appears
  • provider total changes semantics
  • stream final event shape changes

Expected response

  • raw_usage_missing
  • provider_total_mismatch
  • unverified_additivity
  • fixture/test failure
  • explicit allowlist update with reason
Silent wrong totals are worse than loud incomplete data. The architecture should fail closed.
Adding Providers

A new provider should earn the right to affect totals.

  1. Identify provider, API surface, and documented usage shape.
  2. Create realistic simulated fixtures that include the full response wrapper.
  3. Implement the adapter for response and stream usage if relevant.
  4. Map every documented token field to a token type or explicit ignored reason.
  5. Register additivity for each provider/token pair.
  6. Keep uncertain fields unverified until semantics are proven.
  7. Reconcile provider total against sum(quantity_in_total).
  8. Add drift tests for renamed, missing, and partial usage.
  9. Add export and trace-rollup expectations.
  10. Add a real payload slot for later ground-truth capture.
Testing Strategy

Tests should prove contracts, not implementation luck.

Model tests

Validate identities, enum boundaries, quantity constraints, serialization, and absence of stored derived fields.

Adapter tests

Use realistic payloads, provider-specific field maps, completeness checks, and drift fixtures.

Lifecycle tests

Cover streaming interruption, timeout, final usage, retry supersession, and non-authoritative events.

Sink tests

Cover repository idempotency, collector retries, export summability, and privacy leakage audits.

The strongest evidence is a realistic fixture that normalizes, reconciles provider total, and proves every non-additive field contributes zero.

What the architecture is designed to prevent.

Counting mistakes

Adapter Implementation Template

Every adapter should follow the same exact extraction pattern.

class ProviderSurfaceAdapter(BaseAPISurfaceAdapter):
    provider = "provider_name"
    api_surface = "surface_name"

    def extract_usage_from_response(self, response):
        usage = find_usage_object(response)
        model = find_model(response)

        if not usage:
            return NormalizedUsage(
                provider=self.provider,
                api_surface=self.api_surface,
                model=model,
                data_quality_flags=["raw_usage_missing"],
            )

        quantities = []
        quantities.append(self.build_quantity(TokenType.INPUT, usage.input, EXACT, PROVIDER_RESPONSE))
        quantities.append(self.build_quantity(TokenType.OUTPUT, usage.output, EXACT, PROVIDER_RESPONSE))

        return NormalizedUsage(
            provider=self.provider,
            api_surface=self.api_surface,
            model=model,
            quantities=quantities,
            provider_total_tokens=usage.total,
            raw_usage=usage,
        )
The exact field names change by provider. The adapter contract does not.

Unknown Is Not Zero

Missing token data should be visible as uncertainty, not converted into a confident zero.

Bad implementation

Provider omits output count. Code stores quantity=0. Reports show no output cost and no warning.

Correct implementation

Store quantity=null, precision_level=unknown, and an unknown_reason. Contribution is zero but warning remains visible.

Why this is better

Dashboards can separate true zero usage from missing telemetry, and reliability metrics can track unknown quantity rates.

Stream Consumer

The stream consumer is the glue between provider stream events and the stream tracker.

What it does

  1. iterate provider stream events
  2. extract text deltas for fallback estimation
  3. ask adapter whether the event contains usage
  4. keep latest final quantities by token identity
  5. complete with exact usage when output usage is seen
  6. interrupt if stream errors or ends without final output usage

Provider nuance

Some providers split stream usage across events. The consumer should merge final quantities by token type, role, and subtotal parent so the terminal event is coherent.

Collector API

The HTTP collector is small by design and follows the collector client's ack contract.

Route Behavior Important constraints
GET /healthz returns status ok used for liveness checks
GET /v1/stats returns event count, total, totals by trace requires auth when configured; totals are derived
POST /v1/events accepts one event dict or list of event dicts valid items are accepted even if some batch items are malformed
The response is {"acked": [event_ids]}. The client removes only acknowledged ids and retries the rest.

Why Not Simpler?

Why not just sum total_tokens?

Problem

Provider totals exist at event grain and do not understand retries, supersession, non-authoritative observations, partial stream estimates, or multi-provider trace structure.

Architecture answer

Store provider totals as reconciliation facts. Aggregate only contribution-aware event totals, which already handle supersession and authority.

provider_total_tokens: "does this one event reconcile?" event_contributing_tokens: "does this event count in the trace?"
Why Not Simpler?

Why not store totals directly?

Stored totals rot

If a provider semantic changes or a bug is found, stored totals become stale or wrong.

Rules evolve

Additivity, unknown handling, and supersession logic may improve as real payload evidence arrives.

Facts endure

Raw token facts plus explicit categories can be reinterpreted safely by newer derived logic.

Exports can materialize totals for convenience. Storage should keep facts.

Why Not Simpler?

Why not infer additivity from token type names?

Because provider semantics differ

"Cache" is a subtotal for OpenAI but an additive bucket for Anthropic. "Thinking" contributes for Gemini, while "reasoning" can be a subtotal for OpenAI.

Provider plus token type is the key

Additivity must come from a provider-specific truth table. Unknown provider/type combinations must be unverified and excluded until reviewed.

Token type says what the tokens are. Additivity says whether they count.
Why Not Simpler?

Why not include pricing in the tracker?

Prices change

Provider pricing is temporal, model-specific, account-specific, and often region- or tier-dependent.

Token truth is separate

The tracker should produce reliable usage facts. A pricing system can consume those facts later.

Audit clarity

Separating usage from price avoids mixing measurement errors with business-rate decisions.

The tracker can power cost analysis, but it should not be the source of pricing policy.

Tracking HTTP Services

For multi-service systems, context propagation is what connects the token events.

Upstream service

  1. opens root trace or current span
  2. injects X-TokenTracker-* headers into outbound HTTP request
  3. does not need to know downstream provider calls

Downstream service

  1. extracts tracker headers
  2. opens child span
  3. tracks provider calls under same trace
  4. flags propagation_lost if headers are broken
This gives one trace across services without sharing in-memory context between processes.
Tracking Streaming Services

Streaming services track one terminal event, not every text delta as a token event.

Complete stream

Final provider usage arrives. The terminal event uses exact provider input/output quantities and reconciles to provider total if present.

Interrupted stream

No final usage arrives. The terminal event uses a local output estimate and flags partial_stream_estimate and stream_interrupted.

Timeout

The missing quantity is stored as unknown with a reason. It contributes zero but remains visible in quality metrics.

Streaming metrics should include exactness and interruption rate, not only token totals.

Operating Manual

End-to-end lifecycle: from request to audit evidence.

  1. A user, job, or service starts a workflow and creates a trace_id.
  2. The application opens a span for the current step: RAG retrieval, generation, agent planning, tool use, export, or embedding.
  3. The service calls a provider directly, through a proxy, or through a library wrapper.
  4. The provider returns usage in a provider-specific response, stream final chunk, header, or local log.
  5. The adapter translates provider fields into neutral quantities and provider totals.
  6. The normalizer assembles a TokenEvent with identity, quantities, flags, and observation metadata.
  7. The event is appended locally or sent to a collector.
  8. Rollups compute contribution-safe totals from source facts.
  9. Dashboards and exports show totals with quality context.
  10. Audit can trace a dashboard number back to event IDs, fields, rules, and exclusions.

Playbook Provider Mismatch

When provider total and contribution sum disagree, debug semantics first.

  1. Filter mismatches by provider, model, API surface, adapter version, and date.
  2. Inspect one raw payload fixture and the normalized quantities.
  3. Check whether a field is missing, duplicated, incorrectly additive, or a subtotal.
  4. Verify whether provider documentation changed or a new field appeared.
  5. Add or update fixture tests before changing additivity.
  6. Ship adapter version update and annotate affected report period.
  7. Recompute derived rollups from stored source facts.
Do not patch dashboards to hide mismatch. Fix or classify the semantic cause.

Playbook Collector Offline

Collector failure should degrade telemetry delivery, not business execution.

  1. Check collector health endpoint and storage write permissions.
  2. Check auth token mismatch between client and collector API.
  3. Inspect client pending buffer, retry count, and dropped count.
  4. If pending grows, increase flush frequency or restore collector before buffers overflow.
  5. If events are dropped, mark report period with telemetry loss warning.
  6. Replay buffered events if the client/repository supports it.
  7. After recovery, verify event counts by source service and time window.
The collector must be observable itself. A silent collector failure is worse than an explicit telemetry gap.

Final statement

The intended architecture is a trace-aware token observability pipeline. It translates provider usage into neutral source facts, preserves those facts safely, and derives totals with explicit contribution rules. It is not correct because it records numbers; it is correct because it records what each number means.

The most important sentence in the architecture: totals sum quantity_in_total, not raw quantity and not provider_total_tokens.
